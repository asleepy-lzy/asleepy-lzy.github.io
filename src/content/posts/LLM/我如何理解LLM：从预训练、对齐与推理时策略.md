大语言模型明明只做next-token prediction，但为什么可以表现出理解、推理、规划甚至使用工具的能力？更进一步，为什么针对同一个模型，为什么在参数量不变的情况下，采样方式、prompt或测试策略不同，结果就会不一样。
我的理解是，LLM的能力不能只看本身参数量本身，还需要划分为三个层次：
第一：预训练决定模型能学习到什么潜在模式
第二：对齐决定模型是否能按照人类任务要求组织工作；
第三：推理决定模型预训练学习的这些潜在模式在具体任务中被稳定释放出来；


## **第一部分：总览——理解 LLM 的三层框架**

### **1.1 为什么需要一个“三层框架”**

如果只把 LLM 的能力理解成“参数越大越强”，那么很多关键现象其实很难解释。比如，为什么同样是 next-token prediction，模型却会表现出理解、推理和工具使用的样子？为什么同一个模型，在参数完全不变的情况下，仅仅改变 prompt、采样方式或工具调用流程，就会得到明显不同的结果？

一个更清楚的理解方式，是把 LLM 的能力分成三个层次来看。第一层是 **预训练**，它决定模型从数据中学到了哪些统计模式、知识结构与表示能力；第二层是 **对齐**，它决定模型是否会以“助手”的方式组织这些能力，并更稳定地遵循任务接口；第三层是 **推理时策略**，它决定这些已经存在的潜在能力，能否在具体任务中被充分、稳定地调动出来。

因此，这篇笔记想建立的，不是“某个单独方法为什么有效”，而是一个更统一的理解框架：**预训练决定能力空间，对齐决定行为方向，而推理时策略决定能力释放效率。**

### **1.2 一个统一公式与示意图**

如果把这个三层框架压缩成一个更抽象的统一视角，那么可以写成：

$$
\text{Observed Performance}
\approx
g(\text{Pretraining},\text{Alignment},\text{Test-Time Strategy}).
$$

这个式子的含义是：我们在任务中看到的最终表现，并不是某个单一因素的结果，而是三个层面共同作用的产物。预训练提供了模型“会什么”的底座，对齐塑造了模型“倾向于怎么表现”的行为方向，而 test-time strategy 则决定系统在运行时如何组织、探索、外接与筛选这些能力。

也可以把这条主线进一步写成一条更直观的逻辑链：

$$
\text{Pretraining}
\;\rightarrow\;
\text{Alignment}
\;\rightarrow\;
\text{Test-Time Strategy}
\;\rightarrow\;
\text{Observed Performance}.
$$

换句话说，这篇文章的核心目标不是把 LLM 当成一个静态参数块来理解，而是把它看成一个**能力底座 + 行为校准 + 推理时组织**共同构成的系统。

### **1.3 两个贯穿式例子**

为了让后面三部分不只是概念堆叠，可以先看两个会在全文中不断出现的直观例子。

第一个例子是**同一道数学题**。如果直接用 zero-shot 让模型回答，它可能会很快给出一个看起来合理但并不可靠的结果；如果改用 CoT，它会显式写出中间步骤，因而更容易进入分步求解模式；如果再做 self-consistency，它就能在多条推理路径之间投票，提高最终答案的稳定性；如果进一步接上 code execution，那么一部分精确计算就不再依赖模型“心算”，而被交给外部执行器完成。这里变化的并不是参数本身，而是系统在 test time 如何组织解题过程。

第二个例子是**同一个事实性问答任务**。如果只依赖参数内记忆，模型可能会给出流畅但不完全可靠的回答；如果引入 retrieval，再让模型基于检索到的外部文档作答，那么回答的依据就从“参数中的统计记忆”转向了“参数 + 外部证据”的组合。这里变化的也不是模型学到了新知识，而是系统把知识查询这一步外部化了。

这两个例子分别对应后面会展开的两条主线：一条是**如何通过 prompt、CoT、sampling 与 verification 改变推理路径**，另一条是**如何通过 retrieval、code execution 与 tool use 改变系统能力边界**。

有了这个总览之后，后面的三大部分就可以分别回答三个更具体的问题：模型到底学到了什么，模型为什么更像一个助手，以及模型在运行时又是怎样被组织起来解决问题的。


## **第二部分：预训练——LLM 能力的底座从哪里来**

### 2.1 next-token prediction 与自监督学习

大模型的基本任务看起来似乎很简单：给定前面的文本，让模型预测下一个 token。也就是说，如果输入文本是一段前缀文本 $x_1,x_2,\dots,x_{t-1}$，模型学习预测下一个 $x_t$ 的概率分布，整个序列的概率可以写成一连串条件概率的乘积。

大模型的基本任务可以建模为：设一段文本序列 $x=(x_1,x_2,\dots,x_T)$，其中每一个 $x_t$ 都是一个 token。语言模型希望学习到这段文本序列的联合分布 $p_\theta(x)$。根据链式法则，任何联合分布都可以分解为一串条件概率的乘积：

$$
p_\theta(x_1,\dots,x_T) = \prod_{t=1}^{T} p_\theta(x_t\mid x_{<t}).
$$

其中 $x_{<t}=(x_1,\dots,x_{t-1})$。这就是 **next-token prediction** 的数学核心：在第 $t$ 步，模型根据前缀 $x_{<t}$ 来预测下一个 token $x_t$。

如果从对数似然角度来看，训练目标通常为最大化训练语料的似然，等价于最小化负对数似然：

$$
\max_\theta \sum_{x\in\mathcal D}\log p_\theta(x) \quad\Longleftrightarrow\quad \min_\theta \mathcal L(\theta).
$$

其中

$$
\mathcal L(\theta) = -\sum_{x\in\mathcal D}\sum_{t=1}^{T} \log p_\theta(x_t\mid x_{<t}).
$$

再除以 token 总数，这就是常见的平均交叉熵损失。也就是说，训练时模型在每一个位置都要回答同一个问题：**给定前文，真实下一个 token 的概率能不能尽量大。**

从实现的角度来看，模型先会把输入 token 序列映射为隐藏状态，再输出对词表 $V$ 上每个候选 token 的分数。若第 $t$ 个位置的隐藏表示记为 $h_t$，输出层参数记为 $W,b$，那么 logits 可以写成：

$$
z_t = Wh_t + b \in \mathbb R^{|V|}.
$$

经过 softmax 后得到条件概率分布：

$$
p_\theta(x_t=j\mid x_{<t}) = \frac{\exp(z_{t,j})}{\sum_{k\in V}\exp(z_{t,k})}.
$$

对真实 token $x_t$ 的损失就是

$$
\ell_t(\theta) = -\log p_\theta(x_t\mid x_{<t}).
$$

全序列损失则是

$$
\mathcal L_x(\theta)=\sum_{t=1}^T \ell_t(\theta).
$$

这套训练方式成为 **自监督学习**，关键在于监督信号不是人工额外标注的，而是由数据本身自动构造出来的。给定一段原始文本 $x_1,\dots,x_T$，就可以直接生成 $T-1$ 个训练对：

$$
(x_1)\to x_2,\quad (x_1,x_2)\to x_3,\quad \dots,\quad (x_1,\dots,x_{T-1})\to x_T.
$$

也就是说，输入和“标签”都来自同一个未标注文本序列。没有人需要手工告诉模型“这句话的真实续写是什么”。因此，自监督学习不是没有监督，而是监督方式由数据内部结构自动生成。

这里还需要区别 token 和自然语言的“词”。模型并不直接对“单词”建模，而是先通过 tokenizer 把文本离散成 token 序列。设 tokenizer 为映射：

$$
\tau:\mathcal X\to V^*
$$

其中 $\mathcal X$ 是原始文本空间，$V$ 是词表，$V^*$ 表示 token 序列集合。于是，一段原始文本 $s$ 会先变成

$$
\tau(s)=(x_1,\dots,x_T).
$$

模型实际处理的是这个离散序列。token 可以是词、字、符号或字符片段，大语言模型的基本工作单元是 token。这个细节虽然看起来底层，但它直接影响上下文长度、建模效率以及不同语言下的表达能力。

从训练到生成，模型使用的是同一套概率分解框架。训练阶段，模型看到真实前缀 $x_{<t}$，学习 $p_\theta(x_t\mid x_{<t})$。生成阶段则改为递归式地采样或选择：

$$
\hat x_t \sim p_\theta(\cdot\mid \hat x_{<t}).
$$

或在贪心解码中取

$$
\hat x_t = \arg\max_{j\in V} p_\theta(j\mid \hat x_{<t}).
$$

然后把新生成的 $\hat x_t$ 拼接回上下文，再继续下一步。因此，所谓 autoregressive generation，本质上就是把训练时学到的条件分布不断滚动应用。训练是在拟合 $p_\theta(x_t\mid x_{<t})$，推理是在使用这个条件分布。

### 2.2 为什么语言建模会长出“理解”和“推理”的样子

一个更值得关注的问题是：为什么这样一个看似简单的目标，最后会带来如此强的表现？从数学上看，模型虽然只是在最小化 $-\sum_t \log p_\theta(x_t\mid x_{<t})$，但要让这个损失尽可能小，模型必须从前文中提取对预测有用的信息。设某个位置的下一个 token 为 $x_t$，它往往依赖于前缀中的多种因素，例如：

- 局部语法约束；
- 长距离指代关系；
- 语义一致性；
- 事实共现模式；
- 常见问答模板；
- 解题步骤的书写习惯。

换句话讲，next-token prediction 不是一个简单的“补词游戏”。若模型容量足够大、训练数据足够丰富，那么最优地逼近条件分布 $p(x_t\mid x_{<t})$，就要求模型捕捉语言中的大量统计结构。形式上，它学习的是文本分布；现象上，它会表现出对句法、语义、知识乃至某些推理轨迹的建模能力。

还可以从条件熵角度理解这一点。若真实数据分布记为 $p^*$，则在理想情况下，语言模型要逼近的目标与最小化条件熵相关：

$$
H(X_t\mid X_{<t})=-\mathbb E_{p^*}\big[\log p^*(X_t\mid X_{<t})\big].
$$

前缀越能提供信息，条件熵就越低；模型若能有效利用这些信息，其预测也会越准。因此，训练一个语言模型，本质上是在逼迫模型学会如何从上下文中提取对未来 token 最有用的统计信息。

不过，这里也必须明确它的边界。首先，语言模型学习的是文本分布 $p_\theta(x)$，而不是外部世界的真值函数。它优化的是“哪种 token 在这种上下文下更像真实语料中的延续”，而不是“该结论在逻辑上是否一定正确”。因此，$p_\theta(x_t\mid x_{<t}) \text{ 高}$ 并不等价于 $x_t \text{ 必然为真}$。这也是为什么模型可以生成形式上很流畅、局部上很合理、但事实上错误或逻辑不严密的内容。

其次，next-token prediction 所诱导的能力本质上是统计性的，而不是天然形式化证明式的。模型可能学到许多“像推理一样”的文本模式，例如“先列条件，再分步计算，最后给答案”，但这并不自动保证每一步都满足严格的演绎规则。也正因如此，后续的对齐、CoT、self-consistency、search、verifier 等方法才会变得重要：它们并不是替代预训练，而是在这个统计建模底座上进一步提高行为可用性与结果可靠性。

### 2.3 Transformer 与上下文建模

如果说前两节分别回答的是“模型的优化目标是什么”和“为什么这种优化结构可以诱导出模型具有理解和推理的能力”，那么这一节要回答的问题就是：**模型究竟依靠什么结构，才能真正把前文 $x_{<t}$ 转化为对当前 token $x_t$ 有用的表示。** 语言模型的训练目标虽然写成了条件概率 $p_\theta(x_t\mid x_{<t})$，但这个式子本身并没有说明，模型内部究竟如何处理上下文。Transformer 的意义就在于，它提供了一种可扩展的方式，使模型能够在较长上下文中动态选择、组合并重写信息，从而构造出用于预测下一个 token 的隐藏表示。

最经典的模型是 Vaswani 等人在《Attention Is All You Need》中提出的 Transformer 架构。若按照原论文中最经典的总流程图来看，Transformer 可以被理解为一个由 **输入表示、位置编码、编码器、解码器以及最终输出层** 组成的分层信息处理系统。它的核心思想在于：不再依赖循环结构沿时间步递推信息，而是通过 attention 机制，让每个位置都能够直接与序列中相关的位置发生交互，从而更高效地完成上下文建模。

从最底部开始，Transformer 的输入首先是离散 token 序列。由于神经网络不能直接操作符号，模型需要先通过 embedding 层把每个 token 映射为连续向量。设输入序列为 $x=(x_1,x_2,\dots,x_T)$，嵌入矩阵为 $E\in\mathbb R^{|V|\times d}$，则第 $t$ 个 token 的向量表示写为 $e_t = E[x_t]$。这样，原本的离散符号序列就被转换成了向量序列 $(e_1,e_2,\dots,e_T)$。

不过，仅有 token embedding 还不够，因为 attention 本身只关注表示之间的匹配关系，并不天然感知顺序。换言之，如果不额外注入位置信息，那么模型虽然能够知道“有哪些 token”，却不知道“它们按什么顺序出现”。而语言是强顺序结构，同样的词以不同顺序排列，可能产生完全不同的句法和语义。因此，原始 Transformer 在 embedding 之后会加入 positional encoding，得到

$$
h_t^{(0)} = e_t + p_t.
$$

这里的 $p_t$ 表示位置 t 对应的位置编码。在原论文中，位置编码使用的是正弦—余弦形式，它的目的是让模型在没有循环和卷积的情况下，仍然能够感知序列中的相对和绝对位置信息。

在经典总图中，左侧是 Encoder，右侧是 Decoder。Encoder 的作用，是把输入序列编码为一组上下文化表示。每个 encoder layer 都包含两个核心子模块。第一个是 multi-head self-attention，第二个是 position-wise feed-forward network，中间与外侧都带有 residual connection 和 layer normalization。若记某一层输入为 $H$，则 self-attention 的基本计算形式为

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt d}\right)V
$$

其中

$$
Q = HW_Q,\qquad K = HW_K,\qquad V = HW_V.
$$

它的含义是：对于序列中的每个位置，模型都会生成一个 query 向量，并用它去和所有位置的 key 向量计算相关性，再依据这些相关性分数对所有 value 向量加权求和。于是，一个位置的新表示不再只是自身的局部函数，而是整个输入序列信息的加权组合。若只看第 $t$ 个位置，则其 attention 输出可以写成

$$
h_t^{(\mathrm{attn})} = \sum_i \alpha_{t,i}v_i, \qquad \alpha_{t,i} = \frac{\exp(q_t^\top k_i/\sqrt d)}{\sum_j \exp(q_t^\top k_j/\sqrt d)}.
$$

这意味着，encoder 中的 self-attention 允许每个位置直接“看到”整个输入序列，并根据内容相关性选择性地聚合信息。也正是在这一意义上，Transformer 把上下文建模从“沿时间逐步传递”改成了“在全局范围内直接选择”。

之所以称为 multi-head attention，是因为这一过程并不是只做一次，而是并行地做多次。每个头都有自己独立的投影矩阵，因此可以在不同表示子空间中学习不同类型的依赖关系。其形式为

$$
\mathrm{MultiHead}(H) = \mathrm{Concat}(\mathrm{head}_1,\dots,\mathrm{head}_h)W_O.
$$

多头机制的作用在于增强模型对复杂结构的表达能力。因为真实语言中的依赖关系并不单一：既有局部搭配，也有长程指代；既有句法结构，也有语义关联。多个头可以从不同角度并行处理这些关系，再把结果合成为更丰富的上下文表示。

在 self-attention 之后，每个 encoder layer 还会接一个前馈网络：

$$
\mathrm{FFN}(h)=W_2\,\sigma(W_1 h+b_1)+b_2.
$$

这个模块与 attention 的作用不同。attention 负责在不同 token 之间进行信息交互，而前馈网络则逐位置地对表示进行非线性加工。可以说，attention 更像是在“读上下文”，而 FFN 更像是在“加工上下文”。两者交替叠加，使得输入表示在多层 encoder 中被逐步改写，最终形成一组高度上下文化的隐藏状态。这些隐藏状态可以被理解为输入序列的语义表示，也是 decoder 后续生成时要参考的“记忆”。

接下来，在原始 Transformer 的右半部分，decoder 开始工作。Decoder 的结构与 encoder 类似，但它比 encoder 多了一个关键子模块：**encoder-decoder attention**。这也是那张经典总图最重要的特征之一。Decoder 的每一层通常包含三个子模块。第一个是 masked multi-head self-attention，第二个是 encoder-decoder attention，第三个是前馈网络。

先看 decoder 的第一个子模块，即 masked self-attention。它与 encoder 的 self-attention 数学形式相同，但有一个重要区别：这里不能让当前位置看到未来位置，因为 decoder 的目标是逐步生成输出序列。假设输出序列为 $y=(y_1,y_2,\dots,y_T)$，则在预测 $y_t$ 时，模型只能利用 $y_{<t}$，而不能利用 $y_t$ 之后的 token。

为此，decoder 的 self-attention 会加入一个 mask，把未来位置全部屏蔽掉。这样，当前位置的注意力权重只会分配给自己及其之前的位置。这个因果约束保证了模型生成时的自回归性质，也就是

$$
p(y)=\prod_t p(y_t\mid y_{<t},x).
$$

其中 $x$ 是输入序列。这里的 mask 并不是一个次要细节，而是 decoder 能够逐步生成序列的核心条件。

在 masked self-attention 之后，decoder 会执行 encoder-decoder attention。这一步正是原始 Transformer 区别于单纯语言模型结构的重要部分。它的思想是：decoder 当前已经根据先前输出构造出一个中间表示，但如果想生成与输入相关的内容，就还必须去“查询” encoder 提供的输入表示。于是，在 encoder-decoder attention 中，query 来自 decoder 当前状态，而 key 和 value 来自 encoder 输出。若把 encoder 最终输出记为 $H^{enc}$，decoder 当前中间状态记为 $H^{dec}$，则这一步可以形式化为

$$
Q = H^{dec}W_Q,\qquad K = H^{enc}W_K,\qquad V = H^{enc}W_V.
$$

再代入 attention 公式：

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt d}\right)V.
$$

它的含义是：decoder 在生成当前 token 时，不仅会参考自己已经生成的历史，还会回头去输入序列中查找最相关的部分。这一步对于机器翻译尤其关键，因为输出的每个词都往往与输入句子的某些部分直接相关。也正因如此，原始 Transformer 本质上是一个典型的 sequence-to-sequence 架构：encoder 负责理解输入，decoder 负责在参考输入表示的条件下生成输出。

在经过 masked self-attention、encoder-decoder attention 和前馈网络之后，decoder 顶层会得到每个输出位置的隐藏表示。原始流程图最上方接着是一个线性层和 softmax，用于把隐藏向量映射到词表上的概率分布。若最后一层 decoder 在位置 $t$ 的隐藏表示为 $h_t^{(L)}$，则 logits 为

$$
z_t = Wh_t^{(L)}+b.
$$

对应的输出概率为

$$
p(y_t=j\mid y_{<t},x) = \frac{\exp(z_{t,j})}{\sum_{k\in V}\exp(z_{t,k})}.
$$

于是，模型就可以在每一步输出一个词表分布，并进一步选择或采样下一个 token。

Transformer 的整体逻辑可以概括为：底部的 embedding 和 positional encoding 负责把离散输入变成带位置信息的向量序列；左侧 encoder 通过 self-attention 和前馈网络把输入序列编码成上下文化表示；右侧 decoder 一边通过 masked self-attention 建模已经生成的输出前缀，一边通过 encoder-decoder attention 读取输入序列提供的信息；最后顶层的线性层和 softmax 把 decoder 的隐藏状态变成词表上的条件概率。整张图体现的核心思想是，序列建模不再依赖循环，而是完全交给 attention 机制来完成。

不过，对于大语言模型来说，还需要再说明一步：现代多数 LLM 并不直接采用完整的 encoder-decoder 结构，而是主要使用 **decoder-only** 架构。也就是说，它们保留了原始 Transformer 总图中右半部分的自回归生成路径，但去掉了左侧 encoder 和中间的 encoder-decoder attention。原因在于，大语言模型的预训练任务通常不是“输入一句话，输出另一句话”的一般 seq2seq 任务，而是直接在同一串文本上做 next-token prediction。因此，在 LLM 场景中，最关键的是经典图里 decoder 的 masked self-attention、前馈网络、残差与归一化，以及最后的线性层和 softmax。换言之，原始 Transformer 总图提供了一个最完整、最经典的框架，而现代 LLM 则是在这个框架的基础上，沿着自回归生成这一支继续发展出来的。

### **2.4 从目标到能力：训练目标、数据分布与规模的共同作用**


到目前为止，我们已经分别讨论了语言模型的训练目标、它为何会表现出类似“理解”和“推理”的行为，以及 Transformer 如何通过 self-attention 机制完成上下文建模。接下来，一个更整体的问题是：**模型最终学到了什么，究竟由哪些因素共同决定？**

从一个统一视角看，大语言模型的最终能力表现，并不是由单一因素决定的，而更适合写成一种共同作用的结果：

$$
\text{Capability} \approx f(\text{Objective},\text{Architecture},\text{Data},\text{Scale}).
$$

也就是说，模型的表现可以理解为训练目标、模型结构、数据分布与规模效应共同塑造的结果。

首先，**训练目标（objective）** 规定了模型被要求优化什么。对于自回归语言模型而言，这一目标就是通过最小化负对数似然或平均交叉熵损失，逼近真实条件分布
$$
p_\theta(x_t\mid x_{<t}).
$$
它回答的是：模型被要求学什么。从这个意义上讲，预训练阶段并没有显式要求模型学习“知识”“推理”或“规划”，而是统一要求模型更准确地预测真实文本中的下一个 token。

其次，**模型结构（architecture）** 决定了模型如何完成这一目标。Transformer 通过 self-attention 机制，使当前位置能够从整个前缀中动态选择并聚合相关信息，从而构造出有效的上下文表示。它回答的是：模型如何利用前文信息。没有这种高效的上下文建模结构，仅靠目标函数本身并不足以支撑长程依赖、复杂语义和结构化轨迹的学习。

再次，**数据分布（data distribution）** 决定了模型究竟能接触到哪些模式。若训练语料中包含大量事实性文本、问答对话、代码、证明、教程与分步解题过程，那么这些结构都会进入真实数据分布，并成为模型可以学习的对象。它回答的是：模型有什么可学。从这个角度看，大模型表现出的很多能力，并不是由训练目标单独创造的，而是由数据中本来就存在的大量结构化模式支撑的。

最后，**规模（scale）** 决定了模型能把这些模式学到什么程度。模型参数越大、数据越丰富、训练计算越充分，模型对真实分布的逼近通常就越精细，也越有可能稳定捕捉更高阶的语言结构。它回答的是：模型能学得多深、多稳。很多在小模型上并不明显的行为，在更大规模下才会逐渐变得清晰和可复现。

因此，从预训练角度看，大语言模型的能力不能被简单归结为“目标函数好”或者“模型结构强”，而应理解为训练目标、结构机制、数据分布与规模效应共同作用的结果。前面三节分别展开了这些因素中的关键部分，而接下来的 scaling law、in-context learning 与能力涌现，则可以看作这种共同作用在大规模条件下的进一步体现。

### **2.5 Scaling laws、in-context learning 与能力涌现**

在前面几节中，我们已经讨论了语言模型的训练目标、为什么它会表现出类似“理解”和“推理”的行为，以及 Transformer 如何完成上下文建模。接下来，一个更进一步的问题是：**当模型参数更多、训练数据更大、训练计算更充分时，模型究竟会发生什么变化？**

对这个问题，可以从三个彼此相关的现象来理解：**scaling laws、in-context learning 与能力涌现。**

首先来看 **scaling laws**。它并不是某个从理论中严格推出的精确公式，而是从大量预训练实验中总结出来的经验规律：当模型规模、数据规模和训练计算协调增加时，语言模型的损失通常会以比较规则的方式下降。这里的损失通常就是平均交叉熵或负对数似然，即

$$
\mathcal L(\theta) = -\frac{1}{N_{\text{tok}}} \sum \log p_\theta(x_t\mid x_{<t}).
$$

经验上，如果把模型参数量、训练数据量和最终 loss 放到对数坐标中，常常会呈现近似直线关系，因此人们常用幂律形式去拟合它。例如，在只关注模型规模时，可以写成

$$
\mathcal L(N)\approx A N^{-\alpha}+B,
$$

若同时考虑数据规模，则也常写成

$$
\mathcal L(N,D)\approx A N^{-\alpha}+C D^{-\beta}+E.
$$

这里 N 表示模型参数规模，D 表示训练数据规模，\alpha,\beta>0 是通过实验拟合得到的指数，B,E 则表示无法继续降到零的残余项。这个表达式的重点不在于常数本身，而在于说明一个事实：**随着规模增大，模型对真实文本分布的拟合通常会持续改善。**

那么，为什么规模变大，交叉熵往往会下降？本质原因在于，语言模型是在逼近真实条件分布

$$
p^*(x_t\mid x_{<t})
$$

而模型真正给出的只是一个参数化近似

$$
p_\theta(x_t\mid x_{<t})
$$

如果模型太小，它所能表示的函数类就有限，很多复杂依赖关系根本无法容纳，例如长距离指代、复杂格式模式、多层语义约束等。模型变大后，它可表示的条件分布更复杂，就更有可能让 $p_\theta(x_t\mid x_{<t})$ 逼近真实分布 $p^*(x_t\mid x_{<t})$，从而提高对真实 token 的预测概率，降低 $-\log p_\theta(x_t\mid x_{<t})$。

与此同时，更大的数据规模意味着模型能够看到更丰富的事实、表达方式、任务格式和结构化轨迹；更多的训练计算则意味着优化更充分，参数能更接近低损失区域。因此，**模型更大、数据更多、训练更充分**，最终都会体现为交叉熵下降。换句话说，scaling laws 所描述的不是一个神秘现象，而是“模型对真实文本条件分布的逼近随着规模扩大而变得更精细”的经验结果。

如果说 scaling laws 描述的是**优化层面**的连续改善，那么 **in-context learning** 描述的则是这种改善在行为层面的一种具体表现。所谓 in-context learning，指的是：模型在推理阶段不更新参数，仅通过阅读 prompt 中提供的示例，就能够在当前上下文中临时适应一个任务。例如，若在输入中给出若干示例对 $\{(x_i,y_i)\}_{i=1}^k$，再给出一个新的输入 $x$，模型可能直接输出相应的 $y$：

$$
y \sim p_\theta\big(\cdot \mid \mathrm{prompt}(x,\{(x_i,y_i)\}_{i=1}^k)\big)
$$

这里并没有发生梯度更新，参数 $\theta$ 完全不变，但模型却能够根据上下文中的示例调整当前输出模式。

为什么会出现这种现象？一个重要原因是，预训练语料中本来就包含大量“示例—续写”的结构，例如问答、翻译、代码、分类、模板化任务等。模型在长期的语言建模过程中，不只学会了语言内容本身，也学会了这样一种模式：当上下文中已经出现若干输入—输出示例时，后续文本通常会沿着这个任务格式继续展开。因此，few-shot prompt 并不是在真的重新训练模型，而更像是在用上下文给模型提供一个临时的任务说明书。

从表示角度看，这种现象也可以理解为：虽然参数 $\theta$ 没有更新，但 prompt 改变了上下文 $x_{<t}$，从而改变了隐藏表示 $h_t=f_\theta(x_{<t})$，也就改变了模型当前所处的“工作状态”。因此，in-context learning 的本质不是会话内的参数学习，而是模型利用上下文，把任务规则临时写入当前表示之中。

在规模继续扩大时，还会出现第三类现象，即所谓的 **能力涌现（emergent abilities）**。这个词很容易被说得很玄，但它的直观含义其实很简单：某些能力在小模型上几乎看不出来，或者极不稳定；而在模型大到一定程度后，它们突然变得明显、稳定、可用。例如某些多步推理任务、复杂指令跟随任务，可能在小模型上几乎接近随机，但在更大模型上表现会明显改善。

这种“突然会了”的感觉，通常并不意味着目标函数发生了变化。变化的是：随着规模增加，模型开始能够稳定拟合更高阶的语言结构。另一方面，很多外部指标本身就带有阈值效应。例如一道题不是对就是错，只要模型内部表征质量略低于某个阈值，就仍然表现为失败；一旦超过阈值，外部准确率就会突然上升。因此，所谓能力涌现，往往可以理解为两种因素共同作用的结果：一方面，模型确实逐渐获得了表达更复杂结构的能力；另一方面，离散任务指标会把这种连续改善放大成看起来像“突然出现”的现象。

因此，scaling laws、in-context learning 与能力涌现并不是彼此孤立的三个名词，而是一条连续逻辑链在不同层面的体现。scaling laws 说明：随着规模扩大，模型对真实文本分布的拟合持续改善；in-context learning 说明：这种改善会体现为更强的上下文利用与任务适应能力；能力涌现则说明：在某些具有阈值效应的任务上，这种连续提升最终会呈现出非线性的外部表现。换言之，
$$
\text{更大规模} \;\Rightarrow\; \text{更好的分布拟合} \;\Rightarrow\; \text{更强的上下文适应} \;\Rightarrow\; \text{某些任务上出现看似“突然”的能力提升}
$$

### **2.6 预训练阶段的主流范式**


在前面的讨论中，我们主要是从统一框架出发，解释预训练目标、能力来源、结构基础以及规模效应。为了把这一部分整理成更适合复习的结构，还需要进一步回答一个更实际的问题：**在预训练阶段，主流方法到底有哪些，它们分别在建模什么？**

从整体上看，预训练范式并不是杂乱无章的一堆模型名字，而是可以按照“**如何定义预测目标**”来分类。若从这一角度出发，当前主流预训练范式大致可以分为三类：**自回归语言建模（causal language modeling）**、**掩码语言建模（masked language modeling）**，以及**序列到序列建模（seq2seq / conditional language modeling）**。

首先是最重要、也与现代大语言模型最直接相关的 **自回归语言建模**。这一范式的目标，就是前面已经讨论过的 next-token prediction，即将联合分布分解为一连串条件概率：

$$
p_\theta(x_1,\dots,x_T)=\prod_{t=1}^{T}p_\theta(x_t\mid x_{<t}).
$$

它要求模型在每一步只能利用前缀 x_{<t} 去预测当前 token x_t，因此通常需要 causal mask 来保证不能“看到未来”。这一类范式最典型的结构是 **decoder-only Transformer**.

GPT 系列模型就是这一范式的代表。它的优势在于：**训练目标与生成过程完全一致。**训练时预测下一个 token，推理时也逐个生成下一个 token，因此它天然适合开放式生成任务，也成为现代 LLM 的主流基础形式。

第二类是 **掩码语言建模（masked language modeling, MLM）**。这类方法并不是按从左到右生成，而是在输入序列中随机遮掉一部分 token，再让模型根据其余上下文恢复这些被遮盖的位置。若把被遮盖位置的集合记为 M，则训练目标可以写成：

$$
\max_\theta \sum_{t\in M}\log p_\theta(x_t\mid x_{\setminus M})
$$

这里 $x_{\setminus M}$ 表示把被 mask 的位置拿掉之后的可见上下文。与自回归模型不同，MLM 通常允许模型同时利用左侧和右侧上下文，因此更偏向于学习**双向上下文表示**。这一类方法最典型的结构是 **encoder-only Transformer**.

BERT 系列模型是最经典的代表。它更适合理解型任务，如分类、抽取、匹配等，因为它的目标更强调“把整个输入编码好”，而不是“顺序生成一段新文本”。因此，如果说 GPT 类模型更偏向生成，那么 BERT 类模型更偏向表示学习。

第三类是 **序列到序列建模（seq2seq）**，也可以理解为一种条件语言建模。它把输入 x 和输出 y 明确分成两部分，训练目标是：
$$
p_\theta(y\mid x)=\prod_{t=1}^{T_y}p_\theta(y_t\mid y_{<t},x)
$$

在这种建模方式中，模型不是单纯对一个文本序列做续写，而是在条件输入 x 的基础上生成目标序列 y。它通常对应 **encoder-decoder Transformer** 结构：encoder 负责编码输入，decoder 负责在参考输入表示的条件下逐步生成输出。

T5 是这一范式的代表性模型。它的优势在于可以自然统一大量任务，例如翻译、摘要、问答、改写等，因为这些任务都可以表示成“输入一段文本，输出另一段文本”。因此，seq2seq 范式在多任务统一建模方面非常有代表性。

也就是说，不同预训练范式并不只是“模型写法不同”，而是**它们对“上下文如何被使用、输出如何被定义”有不同假设**。


除了建模目标本身，预训练阶段还可以从**训练数据构成**来理解主流路线。现代预训练数据通常不是单一来源，而是多种数据混合。常见类型包括：

- **网页文本（web text）**：提供广泛的语言模式、常识与世界知识；
- **书籍和长文档（books, long-form text）**：提供更长程、更连贯的上下文结构；
- **代码数据（code）**：提供强结构、可执行逻辑和形式化模式；
- **对话与指令数据（dialogue, instruction-like text）**：提供任务接口和交互格式；
- **多模态数据（multimodal data）**：为视觉-语言等更复杂能力打基础

因此，如果把这一节作为复习提纲来使用，那么最值得记住的并不是某一串模型名称，而是下面这个更稳定的分类框架：

- 第一，从**建模方式**看，可以分为 causal LM、masked LM 和 seq2seq LM；
- 第二，从**结构形式**看，可以分为 decoder-only、encoder-only 和 encoder-decoder；
- 第三，从**数据构成**看，可以分为 web text、books、code、instruction-like text 和 multimodal data；
- 第四，从**训练机制**看，需要关注 tokenizer、optimization、long context、stability 与 compute。


换句话说，预训练阶段的主流范式并不是零散术语的堆积，而是一个层次清晰的知识结构：目标定义了预测形式，结构定义了上下文使用方式，数据定义了学习内容，而训练机制决定了这些内容能否真正被学进去。

### **2.7 小结：预训练决定模型学到了什么**

回顾这一部分，可以看到，预训练阶段实际上回答了一个最根本的问题：**大语言模型的能力底座是如何形成的。**

首先，从训练目标上看，现代生成式 LLM 的基本出发点仍然是 next-token prediction。模型通过最小化负对数似然或交叉熵损失，去逼近真实文本中的条件分布$p_\theta(x_t\mid x_{<t})$.这一目标本身并没有显式写入“知识”“推理”或“规划”等概念，但因为真实语言分布中包含了大量语法、语义、事实、结构化轨迹与任务模式，模型在逼近这一分布的过程中，会被迫吸收这些信息。

其次，从能力现象上看，语言建模之所以会长出“理解”和“推理”的样子，不是因为模型突然拥有了一个外加的逻辑模块，而是因为它在优化预测目标时，必须从上下文中提取越来越丰富的统计结构。这种能力本质上首先是统计性的，而不是天然形式化证明式的；也正因如此，模型可以表现出强大的模式生成能力，却仍然可能在事实、逻辑或计算上出错。

再次，从结构机制上看，Transformer 尤其是 self-attention，为这种能力提供了可扩展的实现基础。它使模型能够在较长上下文中动态选择、聚合并重写信息，从而构造出用于预测当前 token 的隐藏表示。换句话说，训练目标回答了“学什么”，而 Transformer 回答了“怎么学”。

然后，从更整体的角度看，模型最终学到什么，并不是由单一因素决定的，而是训练目标、模型结构、数据分布与规模效应共同作用的结果：

$$
\text{Capability} \approx f(\text{Objective},\text{Architecture},\text{Data},\text{Scale}).
$$

其中，objective 决定优化方向，architecture 决定上下文利用方式，data 决定模型可以接触到哪些模式，scale 决定模型能把这些模式学到多深、多稳。

最后，从规模现象上看，随着模型参数、数据规模与训练计算的扩大，模型对真实文本分布的拟合会持续改善，这体现在 scaling laws 所描述的 loss 下降上；而在行为层面，这种改善会逐渐体现为更强的 in-context learning 能力，以及某些任务上看起来像“突然出现”的能力涌现。换句话说，预训练规模扩大带来的，不只是数值上的损失下降，更是模型在行为层面上逐渐显现出更强的上下文适应能力与更高阶的结构化能力。

如果把这一部分压缩成一句最核心的结论，那么可以写成：

> **预训练决定了模型能够从数据中学到哪些模式，以及这些模式在多大程度上能够被稳定表示出来。它并不直接保证模型一定会“正确推理”，但它为后续一切能力——包括知识、上下文适应、生成能力以及推理时策略的发挥——提供了统计与表示层面的基础。**

这也正是为什么，在理解 LLM 时，预训练必须被放在最前面：**它不是模型能力的全部，但它是模型能力的底座。**


## **第三部分：对齐——模型为什么更像“会按要求做事”**

### **3.1 为什么预训练模型还不够**

在第一部分中，我们已经看到，预训练阶段的核心目标是通过 next-token prediction 来逼近真实文本分布，也就是说，模型学习的是

$$
p_\theta(x_t\mid x_{<t}).
$$

这一目标使模型能够从大规模语料中吸收丰富的语言模式、知识结构与任务轨迹，从而形成强大的能力底座。然而，一个自然的问题是：**既然预训练模型已经学到了这么多东西，为什么它还不能直接等同于一个真正“好用”的助手？**

根本原因在于，预训练目标所优化的是**文本分布拟合**，而不是**用户任务完成**。换句话说，模型被要求学习的是“在这样的上下文下，什么样的续写最像真实语料中的延续”，而不是“在这样的用户请求下，什么样的回答最符合人类意图”。这两者并不相同。形式上，预训练关注的是：

$$
\max_\theta \log p_\theta(x),
$$

而一个理想助手真正需要完成的，更像是：

$$
\max_\theta \; \text{Utility}(y\mid x),
$$

其中 x 表示用户输入，y 表示模型回答，\text{Utility} 则衡量回答是否有帮助、是否安全、是否符合要求。显然，前者是分布拟合问题，后者则更接近行为优化问题。

也正因为如此，预训练模型即使已经“会很多”，仍然可能表现出明显的不适合作为助手的行为。例如，它可能无法稳定地遵循明确指令，可能忽略用户要求的格式限制，可能在任务型场景下答非所问，也可能延续训练语料中的网页风格、论坛风格或叙事风格，而不是给出简洁、直接、可执行的回答。甚至在某些情况下，它会继续展开一段“像互联网上会出现的文本”，但这段文本并不真正服务于当前任务。

从一个更直观的角度看，预训练模型擅长的是“把语言续写得像真的一样”，但用户真正需要的是“把任务完成得像助手一样”。这两者之间的差距，主要体现在以下几个方面：

- **指令遵循不足**：模型知道很多东西，但不一定会严格按照“请翻译”“请总结”“请只输出代码”等要求行动。
    
- **格式控制不足**：模型可能输出冗长解释、风格漂移，无法稳定满足结构化输出需求。
    
- **角色行为不稳定**：模型不一定自然维持一个统一的“助手人格”或角色边界。
    
- **偏好与安全未对齐**：模型可能生成技术上流畅但并不有帮助、甚至不适合给用户的内容。
    

因此，预训练所得到的模型，更适合理解为一个**强大的通用语言分布拟合器**，而不是一个已经完成“产品化”的助手系统。它具备潜在能力，但这些能力还没有被整理成稳定、可控、符合任务接口的行为形式。

这也正是为什么需要“对齐（alignment）”阶段。对齐并不是要推翻预训练，而是要在预训练形成的能力底座之上，进一步回答另一个问题：**如何把这些已经存在的潜在能力，变成用户真正可用、可控、可预期的行为？**

最基础的一步，就是用监督方式把模型调整到“指令—回答”的接口上，这就引出了下一节的 **Supervised Fine-Tuning 与 instruction tuning**。

---

### **3.2 Supervised Fine-Tuning 与 instruction tuning**

  

既然预训练模型的主要问题在于“会续写文本”还不等于“会按要求完成任务”，那么最直接的改进思路就是：**显式给模型提供任务输入与理想输出的配对样本，让它学习如何把已有能力组织成符合任务接口的回答。**这就是 **Supervised Fine-Tuning（SFT）** 或更具体地说 **instruction tuning** 的基本思想。

  

在形式上，SFT 并没有改变语言模型的概率建模框架。它仍然是在最大化条件似然，只不过训练数据不再是一般的原始文本，而是更接近任务形式的样本对 $(x,y)$，其中 $x$ 是指令、问题或任务输入，$y$ 是理想回答。对应的目标可以写成：

$$
\mathcal L_{\text{SFT}}(\theta) = -\sum_{(x,y)\in\mathcal D_{\text{inst}}}\log p_\theta(y\mid x).
$$

如果把回答 $y=(y_1,\dots,y_T)$ 再展开，这一目标仍然等价于：

$$
\mathcal L_{\text{SFT}}(\theta) = -\sum_{(x,y)\in\mathcal D_{\text{inst}}}\sum_{t=1}^{T}\log p_\theta(y_t\mid y_{<t},x).
$$

因此，从数学形式上看，SFT 与预训练并没有本质断裂；变化在于：**训练分布从“自然语料续写”切换到了“任务输入—目标回答”这一更具助手属性的分布。**

  

从直觉上看，instruction tuning 做的事情可以理解为：它把预训练阶段已经学到的大量潜在模式，重新组织到一个统一的任务接口上。预训练模型可能已经在网页、教程、问答、代码和对话中见过很多“回答问题”或“执行任务”的例子，但这些能力是分散的、不稳定的，也不一定会在用户需要时被正确触发。instruction tuning 的作用，就是通过高质量的指令数据，让模型逐渐学会一种更稳定的行为模式：

- 当输入是一个任务描述时，要把它理解成“需要完成的请求”；
    
- 当用户指定输出格式时，要优先满足格式约束；
    
- 当输入是问答、翻译、总结、代码生成等任务时，要进入相应的响应模式；
    
- 当存在多种可能回答风格时，要更倾向于选择“助手式”的那一种。
    

  

也就是说，SFT 的核心贡献不在于“创造了新知识”，而在于**把已有能力重新映射到更稳定的任务—输出接口上。**

  

这一步通常会带来非常明显的行为改善。例如，经过 instruction tuning 后，模型往往会更善于：

- 遵循明确指令；
    
- 维持输出格式；
    
- 控制回答长度与风格；
    
- 用更像“助手”的方式回应用户，而不是像网页文本一样漫无边际地续写。
    

  

从这个意义上说，SFT 是对齐阶段最基础、也最不可缺少的一步。它使模型从“通用文本生成器”开始转向“任务响应器”。

  

不过，SFT 也有明显边界。首先，它依赖于人工构造或筛选的指令—回答数据，因此它更像是在已有分布上做行为整理，而不是从零赋予模型全新的底层能力。其次，SFT 优化的是“模仿理想回答”，而不是“根据人类偏好对多个候选回答做细粒度比较”。因此，它虽然能显著提升 instruction-following 和输出稳定性，但还不能充分解决“回答之间哪个更有帮助、更安全、更符合人类偏好”的问题。

  

换句话说，SFT 解决的是“让模型学会按任务接口回答”，但还没有完全解决“在多个可能回答中，哪一个更值得被选择”。这就自然引出了更进一步的问题：**除了让模型学会任务接口之外，我们是否还能让它更稳定地表现出一种“像助手”的交互方式？**这就需要进一步讨论 **chat format、角色设定与任务接口**。

---

### **3.3 Chat format、角色设定与任务接口**

  

如果说 SFT 主要解决的是“模型如何从任务输入映射到目标输出”，那么接下来一个更具体的问题就是：**为什么模型后来越来越像一个聊天助手，而不是一个普通的文本生成器？**

这个问题的答案并不只在参数更新本身，还与输入输出形式的设计密切相关。也就是说，对齐并不只发生在“模型被怎么训练”这一层，也发生在“模型被如何调用”这一层。

  

现代大语言模型的典型交互方式通常采用一种分角色的 **chat format**，常见形式包括：

- **system**：系统级设定，给出总体行为约束；
    
- **user**：用户输入，描述具体请求；
    
- **assistant**：模型输出，承担回答角色。
    

  

从表面看，这似乎只是一个工程协议，但它实际上起到了非常重要的建模作用。因为它把原本开放、模糊的文本续写问题，转换成了一个更明确的交互结构：模型不再只是“继续写下一段文本”，而是在一个角色明确的上下文中，扮演 assistant 去回应 user。换言之，chat format 为模型提供了一种稳定的任务接口，使它更容易进入“我是助手，我现在要回答请求”的行为模式。

  

这种角色结构可以形式化地写成一个带标签的上下文序列：

$$
x = (\texttt{system}, s;\; \texttt{user}, u;\; \texttt{assistant}, ?).
$$

模型需要在前缀

$$
x_{<t} = (\texttt{system}, s;\; \texttt{user}, u;\; \texttt{assistant}, y_{<t})
$$

的条件下继续生成回答：

$$
p_\theta(y_t\mid x_{<t}).
$$

从数学上看，这仍然是条件概率建模；但从语义上看，system、user、assistant 这些显式标签改变了上下文的组织方式，使模型能够更清楚地分辨“谁在说话、当前任务是什么、自己应该以什么身份输出”。

  

在 chat format 中，尤其值得注意的是 **system prompt** 和 **角色设定（role specification）**。system prompt 往往承担全局约束作用，例如：

- 说明模型的身份；
    
- 规定回答风格；
    
- 设定安全边界；
    
- 强调优先级规则。
    

  

因此，它并不只是额外附加的一段文字，而更像是在输入层面加入了行为约束。对于模型来说，system prompt 也是上下文的一部分，会通过

$$
h_t = f_\theta(x_{<t})
$$

影响后续所有隐藏表示和输出分布。也正因为如此，角色设定的本质并不是“魔法控制”，而是利用模型已经学会的上下文敏感性，在输入层面显式塑造其当前工作状态。

  

从更广义的角度看，chat format 可以被理解为一种**统一任务接口协议**。在没有这一协议时，翻译、问答、总结、代码生成、对话等任务可能需要不同的输入形式；而引入 system / user / assistant 结构后，这些任务都可以被统一表示成“用户提出请求，助手给出回应”的形式。这样做的一个重要好处是：模型不再需要在大量不统一的任务格式之间来回切换，而可以在一个较稳定的接口中学习“如何扮演助手”。

  

因此，这一节的关键结论是：**对齐并不只通过 SFT 或 RLHF 发生，也通过 chat format、角色设定与任务接口协议发生。** 参数更新决定模型会不会按某种方式行动，而输入输出协议则决定模型在运行时“被组织成什么样的行为系统”。

  

当然，chat format 和角色设定仍然只是行为组织的一部分。它们能让模型更像助手，但还不能单独回答“在两个同样格式正确的回答中，哪一个更有帮助、更安全、更符合人类偏好”。这个问题需要更细粒度的偏好建模与优化，于是就引出了下一节的 **RLHF 的基本流程与目标**。

---

### **3.4 RLHF 的基本流程与目标**

  

如果说 SFT 让模型学会了“如何按照任务接口回答”，那么接下来更自然的问题就是：**在多个可能回答都看起来合理时，模型应该更偏向哪一种？**

例如，对于同一个问题，模型可能给出更详细但啰嗦的回答，也可能给出更简洁但信息不足的回答；它可能给出技术上正确但不够安全的回答，也可能给出更谨慎的版本。此时，仅靠监督模仿单一参考答案往往不够，因为很多任务并不存在唯一标准答案，而是更接近一种“偏好排序”问题。为了解决这一点，就出现了 **RLHF（Reinforcement Learning from Human Feedback）**。

  

RLHF 的核心思想是：**不用人类直接给每个问题写出唯一正确答案，而是让人类对多个候选回答进行偏好比较，再利用这种偏好信号来优化模型行为。** 典型流程通常包含三个阶段。

  

第一阶段仍然是 **SFT 初始化**。

在这一阶段，先训练出一个基本能对话、能遵循指令的初始模型 $\pi_{\text{SFT}}$。它为后续偏好优化提供一个稳定起点。如果没有这一步，模型生成的候选回答质量可能过低，后面的偏好数据也很难有效收集。因此，RLHF 通常不是替代 SFT，而是建立在 SFT 基础上的进一步优化。

  

第二阶段是 **奖励模型（reward model）训练**。

具体来说，先针对同一个输入 $x$，让模型生成多个候选回答 $y^{(1)},y^{(2)},\dots$，再由人类标注者对这些回答进行排序或成对比较。例如，人类可能认为在输入 $x$ 下，回答 $y^+$ 比回答 $y^-$ 更好。于是可以训练一个奖励模型

$$
r_\phi(x,y),
$$

使它满足：

$$
r_\phi(x,y^+) > r_\phi(x,y^-).
$$

常见的训练方式是采用成对偏好损失，例如 Bradley–Terry 风格的形式：

$$
\mathcal L_{\text{RM}}(\phi) = -\log \sigma\big(r_\phi(x,y^+) - r_\phi(x,y^-)\big),
$$

其中 $\sigma$ 是 sigmoid 函数。这个目标的含义很直观：如果奖励模型能给更受人类偏好的回答更高分，那么它就学会了一个近似的人类偏好函数。

  

第三阶段是 **强化学习优化策略模型**。

有了奖励模型之后，就可以把当前语言模型视为一个策略 $\pi_\theta(y\mid x)$，然后通过强化学习去最大化期望奖励：

$$
\max_\theta \mathbb E_{y\sim \pi_\theta(\cdot\mid x)}[r_\phi(x,y)].
$$

在实践中，常见做法是使用 PPO（Proximal Policy Optimization）等稳定的策略优化算法，同时加入一个 KL 正则项，避免模型过度偏离 SFT 初始策略。于是目标常写成：

$$
\max_\theta \; \mathbb E_{y\sim \pi_\theta(\cdot\mid x)} \Big[r_\phi(x,y)\Big] - \beta \, D_{\mathrm{KL}}\big(\pi_\theta(\cdot\mid x)\,\|\,\pi_{\text{SFT}}(\cdot\mid x)\big).
$$

其中 $\beta$ 控制“追求更高偏好奖励”和“保持原有语言能力稳定”之间的平衡。这个 KL 项非常重要，因为如果没有它，模型可能会为了追逐奖励而偏离原有语言分布太远，导致输出质量崩坏或出现 reward hacking。

  

从目标上看，RLHF 并不是要让模型学会一个全新的世界知识体系，而是要让它在多个可能回答之间，更稳定地朝着人类偏好的方向移动。常见的偏好维度包括：

- **helpful**：回答是否真正有帮助；
    
- **harmless**：回答是否避免明显不安全或不合适内容；
    
- **aligned**：回答是否符合用户请求与系统设定；
    
- **polite / stable / usable**：回答是否更适合作为产品中的助手行为。
    

  

因此，RLHF 可以理解为一种从“模仿理想回答”走向“优化偏好行为”的过程。SFT 更像是在告诉模型“通常应该怎么回答”，而 RLHF 则进一步告诉模型“在多个看起来都能回答的方案里，哪一种更值得被偏好”。

  

当然，这里也需要保持边界感。RLHF 的奖励来自人类偏好，而不是客观真值本身，因此它优化的是“更符合人类偏好”的行为，而不是严格意义上的“真理”。这也就引出了下一节需要专门讨论的问题：**RLHF 到底真正改善了什么，又没有改善什么？**

---

### **3.5 RLHF 做了什么，没做什么**

  

在介绍 RLHF 的流程之后，一个非常重要但也最容易被忽略的问题是：**RLHF 的作用边界到底在哪里？**

如果这个问题说不清，就很容易把 RLHF 误写成一种“万能增强方法”，好像它既能让模型更聪明、又能让模型更安全、还能自动解决事实错误和推理错误。事实上，RLHF 的作用是明确而重要的，但它并没有那么“全能”。

  

首先，RLHF **确实做了很多关键事情**。最直接的一点，是它显著改善了模型的**行为可用性**。如果说 SFT 已经让模型学会了“基本按指令回答”，那么 RLHF 进一步做的是：在很多候选回答中，把模型的行为推向更符合人类偏好的区域。具体来说，它通常会提升以下几个方面：

- **回答更有帮助**：模型更倾向于给出完整、直接、可操作的回答，而不是机械、模糊或跑题的文本。
    
- **行为更稳定**：模型在不同提问方式下，更容易保持一致的助手风格。
    
- **更符合产品需求**：它会更像一个对话助手，而不像一个随意续写网页文本的生成器。
    
- **更注重安全与边界**：对于某些明显不当、危险或不合适的请求，模型更可能采取拒答、规避或重定向策略。
    

  

从这个意义上说，RLHF 的确在很大程度上完成了一个“行为层校准”的工作：它不是重新训练一个全新模型，而是在已有能力底座之上，调整模型把这些能力表现出来的方式。

  

然而，RLHF **没做的事情同样需要说清楚**。

  

第一，RLHF **不等于增加世界知识**。

模型的知识底座主要还是来自预训练阶段接触到的语料分布。RLHF 可以让模型更倾向于以某种方式组织和表达已有知识，但它并没有从根本上给模型注入全新的知识库。若模型预训练时没有见过某些事实，RLHF 本身并不能神奇地补上这一空白。

  

第二，RLHF **不等于保证事实正确**。

奖励模型优化的是“更符合人类偏好的回答”，而不是“严格事实无误的回答”。在很多场景下，人类偏好和事实正确高度相关，但两者并不总是完全一致。模型仍然可能给出流畅、自信、甚至很讨喜的回答，但其中包含事实错误或幻觉。换句话说，

$$
\text{high reward} \not\Rightarrow \text{factually true}.
$$

  

第三，RLHF **不等于直接创造底层推理能力**。

模型是否能进行复杂推理、是否能稳定处理多步任务，底层仍然高度依赖预训练阶段形成的表示与能力结构。RLHF 可以让模型更愿意“解释步骤”“显得更有条理”“表现出更好的任务导向性”，但这并不等于它从根本上发明了新的推理机制。也就是说，RLHF 更多改变的是**行为表现**，而不是**底层智能本体**。

  

第四，RLHF **并不天然避免奖励失真**。

由于它是通过一个奖励模型 r_\phi(x,y) 来近似人类偏好，模型完全可能学会一些“讨好奖励模型”的策略，而不一定真正满足人类真实需求。这类现象常被称为 reward hacking 或 reward misspecification。也正因为如此，实践中才需要 KL 约束、人工评估和后续方法改进，来避免模型过度朝着某种狭窄奖励信号优化。

  

因此，更准确的说法应该是：**RLHF 的核心作用不是重新塑造模型的知识与推理底座，而是对已有模型进行行为层面的偏好校准。** 它让模型更像一个“可用的助手”，但并不意味着模型因此变成了一个在知识、事实和逻辑上都自动可靠的系统。

  

这一点其实也帮助我们更清楚地看待“对齐”的本质。对齐更像是在回答“模型应该怎么表现”，而不是“模型本质上知道多少”。那么，既然 RLHF 是偏好优化的一种经典路线，接下来就自然会有一个问题：**除了 RLHF 之外，是否还有别的方式，也能在不完全依赖 reward model + PPO 这套流程的情况下实现偏好对齐？**这就引出了下一节的 **DPO、RLAIF、Constitutional AI 等扩展路线**。

---

### **3.6 偏好优化的扩展路线：DPO、RLAIF、Constitutional AI 等**

  

在前面的讨论中，RLHF 已经给出了一个非常经典的对齐框架：先做 SFT，再训练奖励模型，然后用强化学习进一步优化策略模型。不过，这条路线虽然有效，也存在明显代价，例如流程较长、实现复杂、奖励模型容易失真、强化学习训练不稳定等。因此，后续研究中逐渐出现了一系列扩展方法。它们的共同目标并没有改变，仍然是：**让模型输出更符合偏好、更像助手、更符合某种行为规范。**不同之处在于，它们在“偏好信号从哪里来”“如何把偏好信号写入训练目标”这两个问题上，采取了不同设计。

  

首先是 **DPO（Direct Preference Optimization）**。

它的核心思想可以概括为：**不再显式拆成“奖励模型训练 + PPO 优化”两步，而是直接利用偏好数据来优化策略模型本身。**

在 RLHF 中，我们先学一个奖励函数 $r_\phi(x,y)$，再让模型朝着更高奖励方向移动；而 DPO 则试图把“更受偏好的回答应当具有更高概率”这件事，直接写进优化目标中。直观上，如果在同一个输入 $x$ 下，人类偏好 $y^+$ 胜过 $y^-$，那么模型就应该满足：

$$
\pi_\theta(y^+\mid x) > \pi_\theta(y^-\mid x).
$$

DPO 的优势在于流程更简洁、训练更直接，也减少了显式强化学习带来的复杂性。对这部分而言，最重要的不是记住它的具体推导，而是知道：**DPO 本质上仍然是偏好优化，只是它绕开了“先学 reward model 再做 PPO”这一经典路径。**

  

第二条重要扩展路线是 **RLAIF（Reinforcement Learning from AI Feedback）**。

顾名思义，它的核心变化在于：偏好信号不再完全依赖人类标注，而是让一个更强模型或辅助模型来生成、评估或筛选反馈。换言之，在 RLHF 中，“HF” 是 human feedback；而在 RLAIF 中，部分 feedback 由 AI 提供。这样做的动机很现实：高质量的人类偏好标注成本高、速度慢、覆盖有限，而模型辅助生成反馈则更容易扩展。

当然，这种方法的优势与风险都很明显。优势在于可扩展性更强，能够快速生成大量偏好样本；风险则在于，如果 AI 自身带有系统性偏差，那么这种偏差也会被进一步放大。因此，从定位上说，RLAIF 仍然属于偏好优化家族，只不过它把“偏好信号的来源”从纯人工扩展到了“人工 + AI”甚至“以 AI 为主”的混合模式。

  

第三条具有代表性的路线是 **Constitutional AI**。

与其说它是在“优化偏好”，不如说它是在“优化原则”。这一路线的核心想法是：与其完全依赖人类对具体答案做大量偏好排序，不如预先给出一套较明确的原则或“宪法”，让模型在此基础上进行自我批评、自我修正和自我改写。于是，模型不只是被动模仿偏好数据，而是开始在一定程度上依据原则来审视和调整自己的回答。

在形式上，这类方法通常包含两个步骤：

第一，让模型生成一个初始回答；

第二，再根据给定原则对回答进行 critique 和 revision。

这意味着，对齐不再只是一种外部奖励驱动，也可以是“原则约束 + 自我修正”的组合。Constitutional AI 的意义在于，它把“行为规范”从隐含的人类偏好数据中，部分转化成了更显式的规则集合。

  

从更高层次看，DPO、RLAIF、Constitutional AI 虽然技术路径不同，但它们在这一部分中的共同位置其实很清楚：**它们都属于“偏好优化 / 行为校准”的扩展路线。**

它们并没有跳出“对齐”这一问题，而是在不同层面上尝试回答同一个问题：

- 偏好信号能否更直接地写进训练目标？
    
- 人类反馈能否部分由 AI 反馈替代？
    
- 对齐约束能否从隐式偏好扩展到显式原则？
    

  

因此，这一节不需要把它们写成各自独立的大综述，而更适合把它们当成 RLHF 家族的自然扩展：**RLHF 是经典主线，DPO 是更直接的偏好优化变体，RLAIF 是偏好来源的扩展，Constitutional AI 则是从偏好走向原则的一种路线。**

  

从博客或复习提纲的角度看，这一节的核心收获不是记住每个缩写的细节，而是理解：**对齐方法正在逐渐从“模仿示范”发展到“优化偏好”，再进一步发展到“引入更灵活的反馈来源与更显式的行为原则”。**

接下来，为了把这一部分整理成更清晰的知识结构，还需要将这些方法压缩成一个更有层次的复习框架，这就引出了下一节的 **对齐阶段的主流方法与关键词**。

---

### **3.7 对齐阶段的主流方法与关键词**

  

在前面几节中，我们已经依次讨论了为什么预训练模型还不够、SFT 如何整理任务接口、chat format 如何固定交互结构、RLHF 如何做偏好校准，以及 DPO、RLAIF、Constitutional AI 等扩展路线。为了让这一部分更适合作为复习提纲，还需要把这些内容进一步压缩成一个更有层次的结构。换句话说，本节要回答的是：**如果只从“对齐阶段的主流方法与关键词”来整理，这一部分最值得记住的是什么？**

  

首先，可以从**训练流程**的角度对对齐方法做一个最基本的分层。

  

第一层是 **监督式对齐（supervised alignment）**，其代表就是：

- Supervised Fine-Tuning（SFT）
    
- instruction tuning
    

这一路线的核心，可以用更自然的话来理解：它关心的是如何让模型真正学会“按指令做事”。也就是说，模型不只是继续生成看起来合理的文本，而是逐渐适应明确的任务格式，学会把输入理解为请求，并以更像助手的方式给出回答。它解决的核心问题是：如何把模型已有能力整理到“任务输入—目标回答”的接口上。

  

第二层是 **偏好驱动对齐（preference-based alignment）**，其代表包括：

- RLHF
    
- reward model
    
- PPO
    
- DPO
    

这一路线更关注“多个回答里哪个更值得选”。换句话说，它不是只让模型学会回答，而是进一步让模型根据人类偏好、奖励建模和策略优化，在多个候选输出中更稳定地偏向那个更有帮助、更安全、也更符合预期的版本。它解决的核心问题是：在多个可能回答中，如何让模型更倾向于输出更符合偏好的一种。

  

第三层是 **反馈来源与原则扩展**，其代表包括：

- RLAIF
    
- Constitutional AI
    

这一路线讨论的是，偏好信号是否一定只能来自人工，以及行为约束是否可以被写成更明确的原则。它强调的不是单纯模仿已有偏好，而是让模型在 AI 反馈、原则约束、自我批评与自我修正的框架下，逐渐形成更稳定的行为边界。它解决的是更进一步的问题：偏好是否只能来自人工？行为约束能否被显式原则化？

  

其次，如果从**输入接口与交互形式**的角度整理，那么对齐阶段同样离不开 chat format、system prompt、role specification 以及 user / assistant / system 这样的角色化结构。它们提醒我们：对齐并不只发生在损失函数层面，也发生在接口协议层面。模型之所以更像一个聊天助手，不只是因为参数更新了，也因为它被放在了一个更稳定的角色结构之中。

  

再次，如果从**优化目标**的角度来看，对齐阶段反复强调的，其实是 helpfulness、harmlessness、honesty、safety 和 alignment 这些行为维度。它们并不对应某个单一算法，而是在提醒我们：对齐真正关心的，不只是输出像不像真实文本，而是它是否对用户有用、对系统可控、并且对风险保持边界。

  

最后，如果从**行为效果**来概括，对齐通常带来的直接结果包括：

- 更强的 instruction following
    
- 更稳定的格式控制
    
- 更一致的对话行为
    
- 更明确的拒答与安全边界
    
- 更像产品中的“助手”
    

  

因此，如果把这一节压缩成复习提纲，那么对齐阶段最值得记住的结构可以概括为：

  

第一，从方法层面看，有 **SFT / instruction tuning → RLHF / DPO → RLAIF / Constitutional AI** 这样一条主线；

第二，从接口层面看，有 **chat format / system prompt / role** 这样一组输入输出协议；

第三，从目标层面看，有 **helpfulness / harmlessness / honesty / safety** 这样一组对齐维度。

  

换句话说，对齐阶段并不是零散技术名词的堆积，而是一套层次分明的知识结构：**SFT 整理任务接口，偏好优化校准行为，角色协议固定交互形式，而最终目标是让模型表现得更像一个稳定、可用、可控的助手。**

---

### **3.8 小结：对齐决定模型如何表现能力**

  

回顾这一部分，可以看到，对齐阶段回答的核心问题并不是“模型还会不会继续学习语言知识”，而是：**模型已经具备的那些潜在能力，究竟会以什么方式被表现出来。**

  

首先，预训练阶段虽然为模型提供了强大的语言、知识和模式能力，但它优化的始终是文本分布拟合，而不是“做一个符合人类需求的助手”。因此，预训练模型即使已经很强，也并不天然保证它会听指令、守格式、维持角色，或者在帮助性与安全性之间做出合适平衡。

  

其次，SFT 与 instruction tuning 提供了对齐的第一层机制。它们通过指令—回答数据，把模型的潜在能力整理到更稳定的任务接口上，使模型开始从“会续写文本”转向“会响应任务”。在这一阶段，模型并没有被赋予完全新的底层能力，而是逐渐学会如何在用户请求下把已有能力组织成可用回答。

  

然后，chat format、角色设定与 system prompt 又进一步补足了交互层面的结构。它们让模型不只是“会回答问题”，而是更稳定地以 assistant 的身份来回答问题。换句话说，对齐并不只是在参数空间中发生，也在输入输出协议层面发生。

  

再往后，RLHF 以及 DPO、RLAIF、Constitutional AI 等扩展路线，则把对齐从“模仿示范”推进到“优化偏好”和“引入原则约束”。这些方法的共同作用，并不是重新构造知识底座，而是进一步校准模型的行为方向，使其在多个可能回答之间更偏向于人类更愿意接受的那一种。

  

因此，如果把这一部分压缩成一句最核心的话，那么可以写成：

  

> **预训练决定模型“会什么”，而对齐决定模型“如何把这些能力表现出来”。**

  

也就是说，对齐并不是重新建造模型的能力底座，而是在已有底座之上，把模型逐步塑造成一个更稳定、更可控、更符合任务接口和人类偏好的系统。它更多改变的是**行为形式**，而不是**能力来源**；更多解决的是“怎么表现”，而不是“凭空创造什么”。

  

这也为第三部分自然铺垫了一个新的问题：即使模型已经被对齐成一个更像助手的系统，**在具体推理时，它的能力是否还能进一步通过 prompting、decoding、sampling、search、tool use 和 verification 等策略被更充分地释放出来？**

这正是下一部分“推理时策略”要讨论的主题。




## **第四部分：推理时策略——为什么不改参数也能显著改变表现**

这一部分讨论的是：**在参数 $\theta$ 固定不变的前提下，为什么仅仅改变 prompt、采样方式、搜索策略、工具调用与验证流程，就能显著改变模型最终表现。**

如果把前面关于预训练与对齐的讨论接起来，可以把这一部分理解为：预训练决定模型学到了哪些潜在模式，对齐决定模型更倾向于以什么方式表现这些模式，而推理时策略决定这些潜在模式能否在具体任务中被稳定、充分地释放出来。

换句话说，模型最终在任务中的表现，并不只是参数本身的直接外显，而还会受到推理阶段组织方式的显著影响。可以把这一点先压缩成一个总公式：

$$
\text{Final Performance}
\approx
f(\text{Prompt},\text{Decoding},\text{Sampling},\text{Search},\text{Tool Use},\text{Verification}).
$$

这并不是说参数不重要，而是说：**参数提供了能力空间，推理时策略决定系统如何在这个能力空间中行走。**

---

### **4.1 什么叫推理时策略**

先要回答的第一个问题是：**什么叫“推理时策略”？**

所谓推理时策略（test-time strategy），指的是：在模型参数 $\theta$ 固定不变的前提下，围绕一次具体任务的输入组织、候选生成、路径选择、外部调用与结果筛选所做的一整套运行时安排。它不发生在训练阶段，而发生在模型已经训练好之后、真正面对任务输入的时候。

从这个角度看，推理并不是一个单一动作，而更像一个由多个环节组成的过程。对于同一个输入 $x$，系统并不一定只能“喂进去然后吐出一个答案”。它还可以决定：prompt 要怎么写、是否给示例、是贪心解码还是随机采样、要不要采样多次、是否显式搜索中间状态、是否调用检索或代码执行工具、以及生成后是否还要做验证与修正。

因此，更一般地说，最终输出其实可以被理解为

$$
\hat y = \mathrm{Infer}(x;\theta,\mathcal S),
$$

其中 $\mathcal S$ 表示推理阶段采用的策略集合。参数 $\theta$ 不变，但 $\mathcal S$ 变化时，$\hat y$ 以及对应的任务表现都可能明显变化。

这一点也解释了为什么 test-time computation 会越来越重要。很多方法并不是通过“重新训练一个更强的模型”来提升表现，而是通过在推理时投入更多计算、组织更多候选、引入更好的筛选机制，让原本已经存在于模型内部的潜在能力被更充分地调动出来。

因此，这一节最关键的结论是：**推理时策略研究的不是“模型学到了什么”，而是“模型在运行时如何被组织起来解决问题”。** 最基础、也最直接的一层组织方式，就是 prompt 本身，这也自然引出下一节的 **Prompting 作为最基本的控制接口**。

---

### **4.2 Prompting 作为最基本的控制接口**

如果说推理时策略研究的是“如何在参数不变时改变表现”，那么最基础的问题就是：**为什么仅仅改写 prompt，就常常能明显改变模型行为？**

最直接的原因在于，对语言模型来说，prompt 不是外围装饰，而就是它所看到的上下文本身。模型在第 $t$ 步的隐藏表示，本质上依赖于此前输入：

$$
h_t = f_\theta(x_{<t}).
$$

因此，当我们改变 prompt 时，真正改变的并不是一句“表面措辞”，而是整个上下文 $x_{<t}$ 的组织形式；而上下文一旦变化，隐藏状态、注意力分配、后续输出分布也都会跟着变化。换句话说，prompting 之所以有效，不是因为模型“被说服了”，而是因为模型的当前工作状态被重新设定了。

这也是为什么 prompt 常常可以起到比想象中更强的控制作用。一个 prompt 不仅能告诉模型“问题是什么”，还会隐式告诉模型“这是什么任务”“应该以什么风格回答”“是否需要步骤化解释”“应该偏向哪类输出格式”。因此，prompting 并不是简单地换一种说法，而是在激活不同的任务模式。

从更抽象的角度看，prompt 可以被理解为最便宜、最通用、最即时的 test-time interface。它不需要额外训练，不依赖外部工具，也不需要引入复杂搜索；只要重写输入，就可能让模型进入完全不同的行为轨道。这也是为什么在实践中，prompt 往往是所有 test-time strategy 的起点。

当然，prompt 的能力也有边界。它能够改变模型对已有能力的调用方式，但并不能凭空创造模型本来没有的知识或计算能力。也正因为如此，我们既要看到 prompt 的控制力，也要保持边界感：它更像是一种“激活与组织机制”，而不是“重新训练机制”。

在理解了这一点之后，下一步自然要问：**prompting 具体有哪些最基本的形式？** 这就进入了 zero-shot、few-shot 与 role prompting 的讨论。

---

### **4.3 Zero-shot、few-shot 与 role prompting**

既然 prompt 是最基本的控制接口，那么接下来更具体的问题就是：**模型最常见的 prompt 级策略有哪些？**

最基础的一种是 **zero-shot prompting**。它的特点很简单：不给示例，直接给任务。例如，输入一个问题、一个翻译请求或一个总结要求，然后让模型直接作答。它依赖的是模型在预训练和对齐阶段已经形成的任务模式，因此本质上是在测试：模型能否仅凭任务描述就进入正确的响应状态。

第二种是 **few-shot prompting**。与 zero-shot 不同，它会在正式任务之前先给出若干输入—输出示例，再给一个新的输入，让模型沿着示例格式继续完成任务。其形式可以写成：

$$
y \sim p_\theta\big(\cdot \mid \mathrm{prompt}(x,\{(x_i,y_i)\}_{i=1}^k)\big).
$$

这里并没有发生参数更新，但上下文中提供的示例改变了模型对当前任务的理解方式。因此，few-shot prompting 与前面预训练部分提到的 in-context learning 是直接相关的：它不是在会话内重新训练模型，而是在利用上下文，把任务规则临时写入当前工作状态之中。

第三种是 **role prompting**。它的核心不在于增加任务示例，而在于通过身份设定改变输出风格与行为边界。例如，让模型扮演“数学老师”“严谨审稿人”“代码助手”，本质上都是在通过角色信息塑造当前的行为先验。它仍然属于上下文控制，只不过控制的重点从“任务格式”转向了“行为身份”。

因此，zero-shot、few-shot 与 role prompting 虽然形式不同，但本质都属于同一类方法：**通过改变上下文组织方式，去影响模型当前被激活的行为模式。**

不过，到这一步为止，prompt 主要还是在控制“回答方式”。接下来更重要的问题是：**能不能让模型不仅回答，而且显式展开中间推理过程？** 这就引出了 reasoning prompting，也就是下一节要讨论的 **Chain-of-Thought、Zero-shot CoT 与 Least-to-Most**。

---

### **4.4 Chain-of-Thought、Zero-shot CoT、Least-to-Most**

普通 prompting 更多是在告诉模型“要做什么任务”，而接下来这类方法关心的问题则是：**能否进一步影响模型“如何展开求解过程”？**

最经典的方法是 **Chain-of-Thought（CoT） prompting**。它的核心思想不是只让模型直接给结论，而是鼓励模型显式写出中间步骤。换句话说，模型不再只是尝试一步到位地产生最终答案，而是把原本可能隐含在表示中的中间推理轨迹，转写成可见文本。

从形式上看，这相当于把原来的

$$
x \rightarrow y
$$

改写成更接近

$$
x \rightarrow z_1,z_2,\dots,z_m \rightarrow y,
$$

其中 $z_1,\dots,z_m$ 表示中间 thought 或分步过程。这样做的关键变化在于：模型不再只输出结果，而开始显式构造一个中间状态序列。

在此基础上，**Zero-shot CoT** 的做法是：即使不给示例，也可以通过类似“Let’s think step by step”这样的触发语句，引导模型进入步骤化生成模式。它说明 reasoning prompting 的一个重要特征：很多时候，决定表现提升的并不是新知识注入，而是是否把中间过程显式化。

**Least-to-Most prompting** 则更进一步。它不只是让模型写步骤，而是先把复杂问题拆成若干更小的子问题，再按从易到难的顺序逐步求解。这意味着推理过程不再只是线性展开，而带有了明显的“先分解、再递进解决”的结构。

不过，这里也必须明确一个边界：**长链条不等于正确推理。** 模型完全可能写出看起来很完整的步骤，但其中某些中间环节仍然是错的、跳跃的，甚至只是风格上“像推理”。因此，CoT 的作用更准确地说，是让中间过程更显式、更可利用，而不是自动保证逻辑正确。

也正因为如此，接下来一个自然的问题就是：即使 prompt 已经确定、推理轨迹也被显式展开了，**模型最终到底沿哪条路径生成答案？** 这就转向了输出层面的控制问题，也就是下一节的 **decoding**。

---

### **4.5 Decoding：greedy、temperature、top-k、top-p、beam search**

到这里，问题的重心开始从“输入如何控制模型”转向“模型如何从输出分布中选择路径”。也就是说：**即使 prompt 相同，模型也未必只有一种生成结果；真正决定它走哪条路径的，是 decoding 策略。**

最简单的 decoding 是 **greedy decoding**。它在每一步都直接选择当前概率最大的 token：

$$
\hat x_t = \arg\max_{j\in V} p_\theta(j\mid x_{<t}).
$$

它的优点是确定、便宜、稳定，但缺点也很明显：一旦某一步局部选择出错，后面整条路径就可能被锁死，而且它天然偏向单一路径。

为了改变这种单一路径偏好，常见做法是引入 **temperature**。若把输出 logits 记为 $z$，则温度采样对应的分布可以写成：

$$
p_T(j\mid x_{<t})=\frac{\exp(z_j/T)}{\sum_k \exp(z_k/T)}.
$$

当 $T<1$ 时，分布会更尖锐，更接近贪心；当 $T>1$ 时，分布会更平滑，模型更容易探索低概率但可能有用的路径。因此，temperature 的本质不是“让模型随机一点”，而是在调节探索与确定性之间的平衡。

在此基础上，**top-k** 与 **top-p** 进一步限制采样空间。top-k 只在概率最高的 $k$ 个 token 中采样；top-p 则选择累计概率达到阈值 $p$ 的那部分候选集合再采样。它们的共同作用，是避免从整个词表中无约束采样，从而在保持一定探索性的同时控制输出质量。

**beam search** 则代表另一种思路：不是每一步只保留一条路径，而是同时保留多个高概率候选前缀，并在后续继续扩展。也就是说，它把单路径选择升级成了有限宽度的多路径维护。

因此，这一节最值得记住的不是每个算法的细枝末节，而是一个统一理解：**decoding 本质上是在做 test-time path selection。** 同一个模型给出的是一个条件分布，而不同 decoding 决定了系统如何从这个分布中选出最终走过的生成路径。

一旦理解了这一点，下一步自然就会出现一个问题：如果单条路径可能偶然走错，那么**能不能一次生成多条路径，再从中聚合出更可靠的结果？** 这就引出了多采样与聚合方法。

---

### **4.6 多采样与聚合：self-consistency、majority vote、best-of-$n$**

在单路径 decoding 中，系统一次只走一条轨迹，因此结果很容易受到局部随机性或局部错误的影响。于是，一个很自然的想法就是：**既然一条路径可能偶然错，那能不能多走几条，再综合它们的信息？**

这就是多采样方法的基本出发点。对于同一个输入 $x$，系统不再只生成一个候选，而是采样得到多个候选回答：

$$
\{y^{(1)},y^{(2)},\dots,y^{(n)}\}.
$$

这样做的核心价值在于提升候选覆盖率。单次采样只看到分布中的一个局部结果，而多次采样则更有机会覆盖不同的 reasoning path、不同的表达形式，以及不同的局部决策组合。

最有代表性的做法是 **self-consistency**。它通常与 CoT 结合使用，让模型对同一个问题生成多条推理路径，再根据这些路径最终是否收敛到同一个答案来做投票。其思想很直观：如果多个彼此独立的 reasoning path 都指向同一个结论，那么这个结论通常更值得信任。

与之相近的还有 **majority vote**、**best-of-$n$** 和各种 reranking 方法。majority vote 直接按最终答案投票；best-of-$n$ 则从多个候选中选一个最好者；如果系统还带有额外评分器或 verifier，那么也可以在采样之后做更细粒度的筛选。

不过，多采样并不是免费午餐。它的收益来自更高的候选覆盖，但代价则是更高的 test-time compute、更长延迟以及更复杂的聚合机制。因此，它本质上是在用更多推理时计算，换取更强的鲁棒性与更高的正确率。

这也说明一个更深的点：**推理能力的表现，不只取决于“模型能不能想到”，还取决于系统愿不愿意给它多次尝试的机会。** 如果再往前推进一步，多采样其实已经在接近搜索了，因为它不再满足于单条路径，而是在探索多条候选轨迹。于是下一节自然就进入 **search-based reasoning**。

---

### **4.7 Search-based reasoning：Tree of Thoughts、beam、MCTS**

多采样已经告诉我们：一条路径不够时，可以多试几次。但接下来更强的问题是：**能不能不只是“多试几次”，而是显式地组织这些尝试，让系统像搜索一样管理中间状态？**

这正是 search-based reasoning 的核心思想。与普通 sampling 相比，search 更强调对中间状态的显式表示、扩展与评估。换句话说，它不只是生成多个最终答案，而是在推理过程中维护一个搜索结构。

从抽象上看，这类方法通常都可以写成一个三元组：**状态（state）—动作（action）—评估（evaluation）**。若记当前状态为 $s_t$，可选动作为 $a_t$，则扩展过程可以理解为：

$$
s_{t+1} = \mathrm{Expand}(s_t,a_t), \qquad a_t \in \mathcal A(s_t).
$$

这里的状态可以是“当前已经写出的 thought 序列”，动作可以是“下一步要扩展的 thought”，而评估则决定哪些分支值得继续保留。

**Tree of Thoughts（ToT）** 的思想就是把 thought 本身当作搜索节点。系统不再只线性生成一串步骤，而是把不同 thought 当作不同分支，在树结构中展开、比较与回溯。**beam search** 可以看作其中较简单的一类做法，即始终保留若干最有希望的分支继续向前。**MCTS（Monte Carlo Tree Search）** 则更进一步，它会引入 rollout、探索与价值估计，使搜索不只依赖局部概率，还考虑未来潜在收益。

不过，这一节最重要的仍然不是算法细节，而是概念上的升级：**reasoning 不再只是“生成过程”，而开始变成“在中间状态空间中的结构化探索”。**

当然，search 仍然主要是在模型内部状态空间中做文章。接下来一个更进一步的问题是：**有些任务是否根本不该只靠模型在内部搜索，而应该把一部分能力外接出去？** 这就进入了工具调用。

---

### **4.8 Tool use：ReAct、PAL、Program-of-Thought、retrieval、code execution**

到这一步，推理时策略已经从 prompt、decoding、sampling 发展到了 search。但一个更现实的问题是：**为什么很多任务不能只靠模型“自己想”，而需要调用外部工具？**

根本原因在于，语言模型虽然擅长模式生成，但并不天然等于一个完备的检索系统、符号计算器或执行环境。因此，在很多任务里，更合理的做法不是要求模型把一切都“脑补出来”，而是让它把某些子任务外部化。

最典型的一条路线是 **ReAct**。它把推理（reasoning）与行动（acting）交替组织起来：模型先形成一个局部判断，再执行一次工具调用，然后根据返回结果继续后续推理。这样一来，系统不再是单次闭门生成，而是形成了“想一想、做一步、看结果、再继续”的循环。

**PAL（Program-Aided Language Models）** 与 **Program-of-Thought** 则体现了另一种外部化思路：把精确计算交给程序。模型负责把问题翻译成代码或可执行过程，而真正的算术、逻辑执行与结果返回则由解释器完成。这样做改变的并不是模型参数，而是系统把“语言组织”和“精确执行”分给了不同模块。

**retrieval** 的作用则是把知识查询外部化。面对事实密集或时效性强的问题时，模型不一定应该只依赖参数记忆，而可以先检索相关文档，再在外部证据基础上回答。类似地，**code execution** 可以把复杂计算、表格处理、程序验证等任务交给执行器完成。

因此，工具调用最值得记住的一点是：**它改变的不是模型内部学到了什么，而是整个系统的能力边界。** 原本只能依赖参数内知识与文本生成的系统，一旦接上检索、程序执行或环境交互，问题求解空间就被重新定义了。

当然，工具并不自动带来更好结果。工具调用需要正确的时机选择、接口设计与结果整合；如果调用错工具、读错返回结果，系统同样会失败。也正因为如此，在生成与行动之后，很多系统还需要一个额外环节：对候选答案进行验证、筛选与修正。这就引出了下一节。

---

### **4.9 Verification 与 refinement：verifier、reflection、self-refine、reranking**

当系统已经能够生成多个候选、甚至还能调用工具后，下一个问题就变成了：**生成出来的结果，为什么还需要再验证和修正？**

原因在于，生成（proposal）与评估（evaluation）并不是同一件事。一个系统完全可能很会提出候选答案，但不一定会在第一次就给出最好的那个。因此，推理时策略不应只理解为“怎么生成”，还应包括“怎么判断生成结果是否可信、是否值得保留”。

从抽象上看，这一类方法往往可以写成“先生成候选集合，再用某种评分器选择”的形式：

$$
y^* = \arg\max_{y\in \mathcal C(x)} v(x,y),
$$

其中 $\mathcal C(x)$ 表示候选集合，$v(x,y)$ 可以是 verifier、reward model、规则评分器，或者模型自身给出的反思性判断。

**verifier** 的作用，就是在多个候选中判断哪个更可信。它未必亲自生成答案，但可以作为一个“第二双眼睛”来做筛选。**reflection** 更强调让模型回头审视自己的回答，主动寻找漏洞、矛盾或遗漏。**self-refine** 则进一步把这种反思变成迭代流程：先给出初稿，再根据反馈改写，再继续优化。**reranking** 的位置则更接近排序器，它负责在一组候选里挑出更优的那个。

这说明一个重要变化：在很多强系统中，“回答问题”已经不再等价于“一次性吐出答案”，而更像是一个闭环流程，包括候选提出、结果评估和必要时的修正。

当然，这里也要看到边界。verification 本身也可能出错，reflection 也可能只是生成更像反思的话语而不是真正发现问题。因此，它提升的是系统的后处理能力与自我校正机会，而不是绝对正确性的保证。

不过，一旦 verification 被引入，我们就必须进一步追问：**到底应该验证什么才算“好”？是只看最终答案，还是连中间过程也要看？** 这就引出了下一节的 process vs outcome。

---

### **4.10 Process vs outcome：过程监督与结果验证**

当推理过程被越来越显式地写出来后，一个新的问题自然出现了：**我们评价一个回答，到底应该只看最后结果，还是还要看中间过程？**

如果只看最终答案，那么这对应的是 **outcome supervision / outcome verification**。它关心的是：最后答对了吗？最终结果是否满足要求？这种做法的优点是直接、明确、易于标注；但缺点是信息较粗，因为它无法区分“过程很好但最后一步失误”和“整个过程从一开始就偏了”。

如果把中间推理链条也纳入评价，那么就更接近 **process supervision / process verification**。它关心的不只是结果是否正确，还关心中间步骤是否合理、是否连贯、是否真正在朝着解题目标推进。随着 CoT、LtM、ToT 等方法的发展，这一点变得越来越重要，因为系统生成的已经不只是答案，还有完整的过程轨迹。

从这个角度看，verifier 也有两种不同角色。它可以只验证最终答案，也可以验证中间过程；而当系统开始做多步推理、搜索或工具调用时，后者往往会变得更关键。因为很多错误并不是只出现在最后一步，而是早在中间状态就已经埋下了。

因此，这一节真正想说明的是：**推理时策略不仅包括“怎么生成”，也包括“按什么标准判断生成得好不好”。** 一旦中间过程被显式化，评价维度本身就会发生变化。

在把这一点说清之后，第三部分的知识结构其实已经比较完整了。接下来，为了更适合作为复习提纲，还需要把这些方法压缩成一个更有层次的总结框架，这就引出了下一节。

---

### **4.11 推理时策略的主流方法与关键词**

回顾前面几节，可以看到，推理时策略并不是零散技巧的堆积，而是可以按照功能位置整理成一个结构清晰的框架。

第一类是 **prompt 级控制**。它包括 zero-shot、few-shot、role prompting、Chain-of-Thought 与 Least-to-Most 等方法。它们共同回答的问题是：在不改参数的前提下，如何通过上下文组织方式，让模型进入更合适的任务模式与推理模式。

第二类是 **输出路径控制**。它主要包括 greedy decoding、temperature、top-k、top-p 与 beam search。这一组方法讨论的是：在模型已经给出条件分布后，系统究竟应当如何从分布中选出一条具体生成路径。

第三类是 **多路径生成与聚合**。这里的代表方法包括 self-consistency、majority vote、best-of-$n$ 以及各种 reranking。它们的共同思想是：单条路径可能偶然出错，因此可以通过多次生成和后续聚合，提高候选覆盖与最终稳定性。

第四类是 **显式搜索**。Tree of Thoughts、beam 式搜索以及 MCTS 都属于这一方向。与普通多采样相比，这一类方法不只是在结果空间里多试几次，而是在中间状态空间中进行有组织的探索与保留。

第五类是 **外部增强策略**。ReAct、PAL、Program-of-Thought、retrieval 和 code execution 都在这一类中。它们的共同点在于：不再要求模型只靠参数内部解决全部问题，而是通过检索、程序执行和环境交互来扩展系统能力边界。

第六类是 **验证与修正策略**。verifier、reflection、self-refine、reranking，以及 process / outcome 这组区分，都属于这个层面。它们强调的是：系统不只是生成答案，还需要对候选进行判断、筛选和必要时的迭代修正。

如果把这些方法进一步压缩成一句话，那么第三部分最值得记住的关键词结构其实就是：**prompting、decoding、sampling、search、tool use、verification，以及它们背后的统一概念 test-time compute。**

换句话说，推理时策略真正研究的，是在参数不变时，系统如何通过更多计算与更好组织，把已有能力转化成更好的任务表现。

---

### **4.12 小结：推理时策略决定能力能否被稳定释放**

回顾这一部分，可以看到，推理时策略回答的核心问题并不是“模型还会不会继续学习”，而是：**模型已经具备的那些潜在能力，在面对具体任务时能否被充分、稳定地调动出来。**

首先，prompting 说明：只要上下文组织方式发生变化，模型当前被激活的任务模式就会变化；few-shot、role prompting 与 CoT 则进一步说明，系统不仅可以控制“回答什么”，还可以在一定程度上控制“如何展开求解过程”。

其次，decoding、多采样与搜索说明：模型输出并不是唯一确定的一条路径。即使参数相同，系统也可以通过不同的路径选择、候选扩展与中间状态探索方式，得到明显不同的结果。很多表现差异，并不来自模型“会不会”，而来自系统“怎么选、怎么试、怎么保留”。

再次，tool use、verification 与 refinement 说明：强系统的推理早已不只是内部文本生成，而是逐步发展成一个更完整的运行闭环，既包括外部检索、程序执行与行动，也包括候选评估、错误修正与结果筛选。

因此，如果把全文中最核心的三层关系压缩成一句最核心的话，那么可以写成：

> **预训练决定模型“会什么”，对齐决定模型“倾向于怎样表现”，而推理时策略决定这些能力能否在具体任务中被充分、稳定、可靠地释放出来。**

也就是说，推理时策略并不是参数学习的附属细节，而是能力释放层。它不重新塑造模型的底层表示，却会深刻影响系统在真实任务中表现出的上限、稳定性与可用性。

如果再把这一部分压缩成一个更抽象的统一视角，那么可以写成：

$$
\text{Observed Performance}
\approx
g(\text{Pretraining},\text{Alignment},\text{Test-Time Strategy}).
$$

其中，预训练决定潜在能力空间，对齐决定行为方向，而 test-time strategy 决定这些能力在任务中的组织、探索、外接与筛选方式。也正因为如此，在理解 LLM 时，我们不能只看“模型参数有多大”，还必须看：**它在推理时是怎样被使用的。**

---

## **结语：把三层放回一个统一视角**

如果把整篇文章再压缩一次，那么最值得记住的并不是某一个缩写、某一个技巧，甚至也不是某一个具体模型名字，而是下面这个更稳定的判断：**预训练决定能力空间，对齐决定行为方向，推理时策略决定能力释放效率。**

这三层之所以重要，是因为它们分别对应了 LLM 系统中的三个不同问题。预训练回答的是“模型从数据中到底学到了什么”；对齐回答的是“模型会不会以符合人类任务接口的方式来组织这些能力”；推理时策略回答的则是“这些能力在真实任务里能否被充分、稳定、可靠地调动出来”。

也正因为如此，很多关于 LLM 的讨论之所以会彼此混淆，往往是因为本来属于不同层次的问题被放在了一起。例如，有些提升其实主要来自更强的预训练底座，有些改善主要是对齐带来的行为变化，而另一些看起来“突然更聪明”的现象，则可能更多来自更好的 test-time organization。只有把这三层拆开，很多现象才会重新变得清楚。

所以，这篇笔记真正想留下的，不只是一个知识清单，而是一种理解方式：**不要只问模型有多大、参数有多少，还要问它学到了什么、被校准成什么样，以及在推理时究竟是怎样被使用的。**

## **延伸阅读**

如果想沿着这篇笔记继续往下读，下面这些材料会是比较自然的延伸起点：

- **Transformer**：Vaswani et al., *Attention Is All You Need*。适合理解 self-attention、encoder-decoder 结构以及现代 LLM 的架构源头。
- **Scaling Laws**：Kaplan et al., *Scaling Laws for Neural Language Models*。适合理解模型规模、数据规模与 loss 改善之间的关系。
- **In-context Learning**：Brown et al., *Language Models are Few-Shot Learners*。适合理解 few-shot prompting 与大模型上下文适应能力的经典起点。
- **Chain-of-Thought**：Wei et al., *Chain-of-Thought Prompting Elicits Reasoning in Large Language Models*。适合理解为什么显式中间步骤会改变推理表现。
- **Self-Consistency**：Wang et al., *Self-Consistency Improves Chain of Thought Reasoning in Language Models*。适合理解多路径采样与投票为什么能提升稳定性。
- **ReAct**：Yao et al., *ReAct: Synergizing Reasoning and Acting in Language Models*。适合理解 reasoning 与 tool use 如何被组织成一个闭环。
- **DPO**：Rafailov et al., *Direct Preference Optimization: Your Language Model Is Secretly a Reward Model*。适合理解偏好优化为什么不一定要经过完整的 reward model + PPO 路线。

如果把这些材料和本文的三层框架对应起来，那么大致可以理解为：Transformer、Scaling Laws 和 in-context learning 更靠近预训练底座；DPO 更靠近对齐机制；CoT、self-consistency 与 ReAct 则更靠近推理时策略。这样继续往下读，整条知识线会更连贯。
